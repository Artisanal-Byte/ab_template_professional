# rules.md

## Operating Mode (Codex)

- Work in increments. No giant one-shot full app implementations.
- After each meaningful change, run the relevant command(s) and verify results (no "assume it worked").
- After verifying results commit and push (if repo connected to an github repo) with a meaningful commit message.
- Keep diffs focused. Avoid unrelated refactors.
- Do not introduce new dependencies unless clearly needed. Prefer built-in framework features first.
- When modifying behavior, update or add tests where reasonable.
- Before adding any new primitive component/functionality not already present in the design system, inform me first and get confirmation.

## MCP Servers

- Laravel Boost: If Laravel Boost is accessible, use it. And `AGENTS.md` file is generated by Boost, you can ignore it for all non-laravel related tasks.

## Code Style (Non-negotiable)

### Naming

- Variables, functions, methods: **camelCase**
- Classes, enums: **PascalCase**
- Files: follow framework conventions (Laravel PSR-4, Vue SFC naming as needed)

### Ternary

- Never use **nested ternary** operators.
- Single-condition ternary is fine.

### Vue / Inertia (Vue 3)

- Use **TypeScript in Vue** (`<script setup lang="ts">`) unless explicitly told otherwise.
- Prefer object-style declarations for:
  - `defineProps({ ... })`
  - `defineEmits({ ... })`
  - `defineModel({ ... })`
    unless types are very complex.
- Avoid multiline functions inside `<template>` directives (`@click`, `@change`, etc.).
  - Extract logic to `<script>` and call it.
  - Single-line lambdas like `(x) => foo(x)` are acceptable.

### PHP / Laravel Types

- Enforce types everywhere practical:
  - `declare(strict_types=1);`
  - typed properties
  - parameter and return types
  - prefer enums/value objects over magic strings where it improves clarity

## Laravel Conventions

- Use Form Requests for validation.
- Prefer thin controllers:
  - controllers orchestrate
  - domain/service classes implement logic
- Use database transactions when writing multiple related records.
- Prefer explicit policies/permissions for auth logic.

## DB (PostgreSQL)

- Prefer correct column types (uuid/jsonb/timestamptz/etc).
- Add indexes for real query paths (foreign keys, lookup columns).
- Keep migrations reversible when feasible.
- For table and column names, prefer descriptive, self-explanatory names (even if they are long).

## Frontend UI (TailwindCSS)

- Use Tailwind utility-first; avoid custom/RAW CSS unless necessary.
- Prefer extracting repeated class bundles into components rather than copying.

## Quality Gates (pick what exists in the repo)

- PHP: run `pint` (and phpstan/larastan if configured)
- JS/TS: run lint + typecheck (`vue-tsc` if configured)
- Run tests relevant to the change set (Pest/PHPUnit/Vitest/etc).

## Safety / Foot-guns

- Do not run destructive commands that can delete data or wipe environments.
- Prefer additive migrations over dropping columns unless explicitly required.

## Package Management Preference

- Prefer `bun` for JS/Node dependency management when a JS package manager is needed.
